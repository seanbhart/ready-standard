# Ready

The Ready Product Tree stores product truth as a filetree. Ready primitives are
typed, addressable records: product logic, Decision flags, artifact descriptors,
governance records, and generated-view inputs live in ordinary files so teams
can branch, review, merge, fork, and compare product decisions with the same
tools they use for code.

A Ready branch represents one coherent intended product state. Git history and
branches carry evolution and hypothetical alternatives; apps may derive
discrepancies by comparing Ready truth to code, evidence, generated views, or
prior commits. Stored flags add durable proof, attention, blocker, and
coordination gates; they are not change instructions.

The standard separates primitive class from compiler authority:

- Product-logic `.ready.yml` primitives describe why the product exists, what
  it promises, how it must behave, and what it depends on.
- X primitives describe temporary process interrupts and proof gates: open
  Decisions, attention, blockers, routing, review, approval, correction,
  escalation, drift, and claim-readiness state.
- A primitives describe durable supporting content: proof corpora, documents,
  resources, snippets, designs, manifests, reports, and generated-view inputs.
  Large or sensitive payload files may live beside the primitive record as
  attachments or safe refs.
- Governance primitives constrain authority, process, and agent behavior.
- Generated views compile from the tree for different audiences.
- Local chats, scratchpads, search indexes, raw logs, and transcripts stay in
  local cache unless intentionally promoted into product records.

## Why this exists

Product direction often drifts because requirements, decisions, tickets, mockups,
chat, and implementation proof live in separate systems. A filetree product model
keeps the durable part small, reviewable, and portable while still letting apps
render richer PM, developer, designer, and customer views.

## Minimum conforming tree

```text
ready/
  manifest.yaml
  governance/
  settings/
  product/
    manifest.yaml
    premises/
    intents/
    standards/
    services/
    artifacts/
      samples/
      resources/
  flags/
    decisions/
    blockers/
    drift/
```

The directory names can vary by product, but the standard requires one durable
record per Ready primitive, stable ids in the record, directed refs where
relationships matter, explicit compiler roles, and explicit proof gates before
coding work can start. Intended product truth lives in `ready/product/`.
Temporary workflow records, discrepancy notes, blockers, and decisions may live
in `ready/flags/`. Git-tracked workspace/app settings may live under
`ready/settings/`.

Reusable Ready skills and templates live in the portable standard package. A
project that expects agents to edit the tree should vendor that package at
`ready/standard/` so agent behavior is pinned and reviewable. The `docs/`
directory in this source repo is Mintlify documentation source; product repos
do not vendor it.

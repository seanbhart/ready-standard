# Ready

The Ready Standard defines `.ready` packages: committed product source trees an
orchestrator reads to build, prove, operate, or govern the intended product.
The `.ready` format is the durable package and source-record identity; this
version uses `.ready.yml` as the YAML serialization for source records.

A `.ready` branch represents one coherent intended product state. Git history
and branches carry evolution and hypothetical alternatives; apps may derive
discrepancies by comparing Ready truth to code, evidence, generated views, or
prior commits. Stored flags add durable proof, attention, blocker, and
coordination gates; they are not change instructions.

`.ready` records use object families:

- `primitive` records are durable package inputs: premises, intents, standards,
  services, and artifact descriptors.
- `flag` records are temporary package-work records for unresolved
  Ready-package decisions, proposed changes, discrepancies, readiness gates, and
  proof gaps.
- `governance` records constrain authority, process, policy, and agent
  behavior.
- `setting` records hold git-tracked workspace or app configuration.
- `manifest` records describe package structure.

## Why this exists

Product direction often drifts because requirements, decisions, tickets,
mockups, chat, implementation proof, and code live in separate systems. A Ready
package keeps the durable part small, reviewable, and portable while still
letting apps render richer PM, developer, designer, and customer views.

## Minimum Conforming Tree

```text
ready/
  manifest.yaml
  standard/
  governance/
  settings/
  product/
    manifest.yaml
    premises/
    intents/
    standards/
    services/
    artifacts/
      samples/
      resources/
  flags/
    decisions/
    blockers/
    discrepancies/
    readiness/
    proof/
```

The directory names can vary by product, but the standard requires stable ids,
typed records, directed refs where relationships matter, explicit compiler
roles, and explicit proof gates before coding work can claim completeness.
Intended product truth lives in `ready/product/`. Temporary package-work records
for proposed changes, discrepancies, blockers, decisions, readiness gates, and
proof gaps live in `ready/flags/`.
Git-tracked workspace/app settings may live under `ready/settings/`, but runtime
artifacts, build outputs, uploaded product data, and product-owned manifests do
not. Ready-carried artifacts may seed the product, but implementation copies
belong in product-owned paths or storage.

Reusable Ready skills and templates live in the portable standard package. A
project that expects agents to edit the tree should vendor that package at
`ready/standard/` so agent behavior is pinned and reviewable. The `docs/`
directory in this source repo is Mintlify documentation source; product repos
do not vendor it.

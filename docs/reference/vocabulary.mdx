# Ready Vocabulary

The canonical vocabulary source is [`vocabulary.yaml`](../../vocabulary.yaml).
It is part of the portable `ready/standard/` bundle and is the source of truth
for tools, validators, app views, and agents.

Project teams may add local display labels, but stored Ready source records
should use the values from `vocabulary.yaml`. Extensions must be explicit,
reviewable, and local to that project; status values remain `ready`, `draft`,
and `blocked`.

## What It Defines

`vocabulary.yaml` defines:

- record kinds and their schemas;
- standard tree directories;
- primitive classes;
- primitive types by class;
- flag classes and flag types by class;
- governance, setting, and manifest classes;
- record statuses, status rules, and compile behavior;
- blocker rules;
- proof corpus lifecycle states;
- compiler roles;
- claim levels;
- structural relationship roles;
- semantic relationship display phrases in both directions.

Primitive vocabulary defines durable bundle inputs: premises, intents,
standards, services, and artifacts. Flag vocabulary names temporary bundle-work
records: unresolved decisions, proposed changes, blockers, discrepancies,
readiness gates, and proof gaps. It does not create stored implementation
instructions; apps and agents derive reader phrases and coding handoffs from the
resolved intended product state.

Flags omit top-level `status` unless they are blocked. The temporary
bundle-work state is implied by `kind: flag`; do not write `status: draft` or
`status: ready` on flags.

Any record with a non-empty blocker list, such as `blocked_by` or
`fields.blockers`, uses `status: blocked` and requires explicit override before
implementation.

Blocker flags also use `status: blocked`, remain non-claimable, and require
resolution or explicit override before implementation.

Proof corpus lifecycle values are `frozen`, `replaceable`, and `retired`. Draft
proof corpora use top-level `status: draft`, not `fields.lifecycle_state:
draft`.

`compiler_role` is a source-record hint for compilers and orchestrators. It does
not replace `kind`, `class`, or `type`; it clarifies how the record should be
used in compiled product, proof, governance, handoff, or workflow views.

## Relationship Display

Source records store only structural `refs` roles: `peer`, `parent`, and
`child`. Reader views derive semantic phrases from the `relationship_display`
entries in `vocabulary.yaml`.

Each semantic display entry must define both `source_to_target` and
`target_to_source` for the same stored edge. For example,
`intent --parent--> premise` renders as the intent `serves` the premise, and as
the premise is `served by` the intent. The lookup keys are based on source
kind, optional source class, target kind, optional target class, and stored
structural role. The kind keys matter for cross-kind refs such as
flag-to-standard relationships because flags are not primitives.

Display entries may use documented wildcard class tokens from
`relationship_display_wildcards`. `any` matches any class for the specified
kind. `same_or_related` matches the same class or a project-approved related
class.

Standards that govern implementation-bearing records normally use `peer` refs:
`intent --peer--> standard` and `service --peer--> standard` render as
`governed by` / `governs`. Premises should not attach standards or services
directly; if a standard or service appears related to a premise, attach it to
the intent that serves the premise. Flags that are governed by standards use the
same `peer` display.

Subordinate intent-to-intent relationships are authored by the parent intent:
`intent --child--> intent` renders as `fulfilled by` / `fulfills`. Lateral
intent-to-intent relationships may use `peer` refs and render as
`coordinates with` in both directions. Do not store an inverse parent ref on the
subordinate intent for the same assertion.

`standard --peer--> standard` renders as `co-governs with` in both directions.
The vocabulary keeps child-standard display phrases as fallback semantics for
older trees and intentionally nested subordinate standards.

Artifact-to-artifact refs may be reciprocal when both edges are independently
meaningful artifact assertions. For example, two artifacts may cite each other
for different evidence, context, or source-material reasons. Do not store a
reciprocal artifact ref merely to mirror the inverse display for one assertion;
reader views derive incoming labels such as `cited by` or `supported by` from the
single authored edge.

## Extension Rule

If a project extends the vocabulary, the extension should be explicit,
reviewable, and local to that project. Do not create new stored relationship
roles when a semantic display phrase can derive from `peer`, `parent`, or
`child`.

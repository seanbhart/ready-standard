# Directory Layout

A product tree should keep records fragmented by scope and type. The root
manifest declares settings, shared scope, and the stage registry; tools should
not infer stage order or default selection from directory names.

```text
ready/
  README.md
  manifest.yaml
  standard/
    manifest.yaml
    skills/
    templates/
  governance/
    agent-guidelines.md
    product-orchestrator-charter.md
    product-process.md
    workspace-authority-and-access.md
  settings/
    product-agent-context.ready.yml
  shared/
    README.md
    manifest.yaml
    premises/
    intents/
    standards/
    services/
    flags/
    artifacts/
  stages/
    end-state/
      README.md
      manifest.yaml
    m1/
      README.md
      manifest.yaml
      premises.md
      premises/
        p-001-example.ready.yml
      intents.md
      intents/
        i-001-example.ready.yml
      standards.md
      standards/
        st-001-example.ready.yml
      services.md
      services/
        sv-001-example.ready.yml
      flags.md
      flags/
        seed/
          xs-001-example.ready.yml
        change/
        discovery/
      artifacts.md
      artifacts/
        manifest.yaml
        samples/
          ap-001-example.ready.yml
          payloads/
            AP-001.json
        resources/
          ar-001-example.ready.yml
        snippets/
          asn-001-example.ready.yml
        designs/
          ad-001-example.ready.yml
        manifests/
          am-001-example.ready.yml
```

`ready/settings/` is not a primitive scope. It holds git-tracked workspace and
app settings that help tools manage the Ready tree, such as product-agent
context preferences, view settings, or local app defaults that should travel
with the repo. Settings files should use explicit settings schemas, avoid raw
secrets, and not duplicate product truth that belongs in primitives,
governance, artifacts, or flags.

`ready/shared/` is not a stage. It holds product truth that applies across all
or multiple stages: enduring standards, reusable services, durable premises,
cross-stage intents, reusable artifacts, and other shared records. This is the
default home for persistent and end-state product definitions, even when some
of that product truth is not implemented yet. Stage records should reference
shared primitives instead of copying them.

Stage-specific truth lives under `ready/stages/<stage-id>/`:

```text
ready/stages/m1/
    premises.md
    premises/
      p-001-example.ready.yml
    intents.md
    intents/
      i-001-example.ready.yml
    standards.md
    standards/
      st-001-example.ready.yml
    services.md
    services/
      sv-001-example.ready.yml
    flags.md
    flags/
      seed/
        xs-001-example.ready.yml
      change/
      discovery/
    artifacts.md
    artifacts/
      manifest.yaml
      samples/
        ap-001-example.ready.yml
        payloads/
          AP-001.json
      resources/
        ar-001-example.ready.yml
      snippets/
        asn-001-example.ready.yml
      designs/
        ad-001-example.ready.yml
      manifests/
        am-001-example.ready.yml
```

Stage directories are best for stage-specific workflow and temporary records:
decisions, seed/change/discovery flags, proof work, and product primitives that
are intentionally scoped to that stage. A stage may own product services,
standards, problems, or solutions when they are temporary, experimental,
historical, or otherwise not durable cross-stage product truth.
Stage kinds such as `horizon`, `experiment`, and `archive` are for stages with
special selection or workflow semantics; they are not needed just to preserve
durable definitions that fit in shared scope.

Top-level plural Markdown files are navigation indexes. Product-logic primitive
source records live inside the directories as `.ready.yml` files. Other
primitive classes can live in their own family directories, such as `flags/`,
`artifacts/`, or `governance/`, with schemas that match their machine contract.
Decision flags are decision/workflow `readyroom/flag/v1` records with
`type: decision`, use `XD-*` ids, and live under `flags/discovery/`.

`ready/manifest.yaml` should include `settings_directory`, `shared_scope`,
`stage_root`, `default_stage`, and a `stages` list with `id`, `title`, `path`,
`manifest`, `kind`, `status`, `order`, `default_candidate`, and
`coding_claims_enabled`. Use `kind: normal` for buildable stages and
`kind: horizon` for `end-state` or other future concept shelves.

Project-specific governance primitives belong under `ready/governance/`.
Reusable Ready resources, task-skill packages, and templates belong in the Ready
standard package. Projects that expect agents to edit the tree should vendor the
portable package at `ready/standard/` and point `ready/manifest.yaml` at it.
That vendored package contains `README.md`, `manifest.yaml`, `skills/`, and
`templates/`; it excludes Mintlify `docs/` and Git metadata.

Artifact descriptors belong under the scope they support. Shared artifacts live
under `ready/shared/artifacts/`; stage-specific artifacts live under
`ready/stages/<stage-id>/artifacts/`. Large or sensitive payloads may be stored
as safe refs. Products may add `compiled/` or `views/`, but generated outputs
must be clearly marked and traceable to source record hashes.

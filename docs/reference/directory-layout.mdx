# Directory Layout

A product tree should keep records fragmented by authority and type. The root
manifest declares product, flags, governance, settings, and standard package
locations. The standard does not define additional product-state directories.

```text
ready/
  README.md
  manifest.yaml
  standard/
    manifest.yaml
    vocabulary.yaml
    skills/
    templates/
  governance/
    agent-guidelines.md
    product-orchestrator-charter.md
    product-process.md
    workspace-authority-and-access.md
  settings/
    product-agent-context.ready.yml
  product/
    README.md
    manifest.yaml
    premises/
    intents/
    standards/
    services/
    artifacts/
      samples/
      resources/
      snippets/
      designs/
      manifests/
  flags/
    decisions/
      xd-001-example.ready.yml
    blockers/
    discrepancies/
    readiness/
    proof/
```

`ready/settings/` is not a primitive scope. It holds git-tracked workspace and
app settings that help tools manage the Ready tree, such as product-agent
context preferences, view settings, or local app defaults that should travel
with the repo. Settings files should use explicit settings schemas, avoid raw
secrets, and not duplicate product truth that belongs in primitives,
governance, artifacts, or flags.

`ready/product/` holds the intended product truth for this Ready tree version:
durable premises, intents, standards, services, product artifacts, proof
corpora, and other records that define what should exist. It is the default
home for product definitions even when some of that product truth is not
implemented yet.

Product truth usually follows this shape:

```text
ready/product/
    premises.md
    premises/
      p-001-example.ready.yml
    intents.md
    intents/
      i-001-example.ready.yml
    standards.md
    standards/
      st-001-example.ready.yml
    services.md
    services/
      sv-001-example.ready.yml
    artifacts.md
    artifacts/
      manifest.yaml
      samples/
        ap-001-example.ready.yml
        payloads/
          AP-001.json
      resources/
        ar-001-example.ready.yml
      snippets/
        asn-001-example.ready.yml
      designs/
        ad-001-example.ready.yml
      manifests/
        am-001-example.ready.yml
```

`ready/flags/` holds temporary package-work records:

```text
ready/flags/
    decisions/
      xd-001-example.ready.yml
    blockers/
      xb-001-example.ready.yml
    discrepancies/
      xdr-001-example.ready.yml
    readiness/
      xr-001-example.ready.yml
    proof/
      xp-001-example.ready.yml
```

Flags are for unresolved Ready-package decisions, proposed changes,
version-divergence notes, blockers, discrepancies, readiness gates, and proof
gaps that prevent the package from being complete, accepted, or safely claimed.
Durable product results belong back in `ready/product/`. A completed package has
no flags. Substantial hypothetical concepts usually belong in alternate Ready
branches or worktrees until accepted into the current branch's intended product
state.

Top-level plural Markdown files are navigation indexes. Primitive source
records live inside `ready/product/` directories as `.ready.yml` files. Flags,
governance records, settings records, and manifests are separate Ready object
kinds with schemas that match their machine contract. Decision flags are
`ready/flag/v1` records with `kind: flag`, `class: decision`, an
appropriate decision `type`, `XD-*` ids, and usually live under
`ready/flags/decisions/`.

`ready/manifest.yaml` should include `product_directory`, `flags_directory`,
`governance_directory`, `settings_directory`, and `ready_standard`. Tools should
not infer product truth, workflow state, or settings from directory names when
the manifest declares explicit paths.

Project-specific governance records belong under `ready/governance/`.
They travel with the Ready package so authority and process versions are shared,
but they are package-management context rather than product runtime
implementation truth.
Reusable Ready resources, task-skill packages, and templates belong in the Ready
standard package. Projects that expect agents to edit the tree should vendor the
portable package at `ready/standard/` and point `ready/manifest.yaml` at it.
That vendored package contains `README.md`, `manifest.yaml`,
`vocabulary.yaml`, `skills/`, and `templates/`; it excludes Mintlify `docs/`
and Git metadata.

Artifact descriptors usually live under `ready/product/artifacts/`. A flag may
own a small workflow-only artifact under `ready/flags/` when it only exists to
support that temporary record. Large or sensitive payloads may be stored as safe
refs. Products may add `compiled/` or `views/`, but generated outputs must be
clearly marked and traceable to source record hashes.

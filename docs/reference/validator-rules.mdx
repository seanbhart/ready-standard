# Validator Rules

A product-tree importer should fail loudly when source truth is ambiguous.

## Required checks

- Every product-logic primitive source record is a `.ready.yml` YAML document.
- Every product-logic primitive source record has
  `schema: readyroom/primitive/v1`.
- Every product-logic primitive source record has `kind: primitive`.
- Every Ready primitive record, including specialized flag, artifact, decision,
  and governance schemas, has a stable `id`.
- No two records share the same `id`.
- Artifact descriptors use `A*` ids. Optional class suffixes may identify
  well-understood artifact classes, but every artifact id must start with `A`.
- Flag records use `X*` ids. Optional class suffixes may identify
  well-understood flag classes, but every flag id must start with `X`.
- Primitive `type` and primitive class are valid.
- Product-logic primitive subtypes are valid according to
  `ready/standard/vocabulary.yaml` when present.
- Product-logic primitives live under the configured `product_directory`.
- Flag records live under the configured `flags_directory` unless a project
  intentionally embeds workflow records beside related product files.
- Settings files live under `settings_directory` when present. They use explicit
  settings schemas and are not primitive source records.
- Constraints, rules, quality bars, compliance needs, measurement rules, and
  non-negotiable behavior are modeled as `standard` records, not premise
  subtypes.
- Structural refs point to existing ids.
- Ref roles are one of `peer`, `parent`, or `child`; semantic verbs such as
  `serves`, `requires`, `governed_by`, `contains_premise`, and `questions` are
  derived reader labels and are not accepted as stored roles.
- Semantic relationship display lookups in `vocabulary.yaml` define both
  source-to-target and target-to-source phrases for the same stored edge.
- Generated indexes do not carry independent primitive truth.
- Committed intents carry concrete `completion_proof` that functions as intent
  Definition of Done, not a one-word status.
- Flag records keep `type`, `claimable`, and `completion_proof` at the top level.
- Decision flags use `type: decision`, stable `XD-*` ids, `fields.question`,
  `fields.safe_default`, structured `fields.proposals`, and concrete top-level
  `completion_proof`.
- Change flags use `type: change` and stable `XC-*` ids.
- Decision-specific content lives under `fields`; top-level keys are reserved
  for shared flag metadata, claimability, blockers, refs, artifacts, owner
  notes, and Completion Proof.
- Seed and change flags are non-claimable unless they pass the claim gate.
- Claimable flags carry concrete Completion Proof that functions as the
  Definition of Done, not a one-word status.
- Claimable coding handoffs resolve to the intended product state; a flag that
  only describes a delta or implementation edit instruction is not enough.
- `ready` flags with non-empty `blocked_by` remain non-claimable.
- Product-logic primitive files do not carry implementation lifecycle state.
- Decision/workflow primitive files do not become coding claims unless their
  gate fields pass.
- Project-specific governance docs are stored under `ready/governance/`, not artifact directories.
- Project-agnostic Ready resources and templates are referenced from the local
  `ready/standard/` package or the Ready standard source package, not artifact
  directories.
- Vendored standard packages exclude Mintlify `docs/`, Git metadata, local
  cache, raw logs, transcripts, private samples, and secrets.
- Artifact descriptor ids are unique within their artifact family and globally
  unique when a project uses global id lookup.
- Artifact descriptors carry the required top-level fields: `schema`, `kind`,
  `id`, `type`, `title`, and `status`.
- Artifact descriptors carry `fields.purpose`, `fields.content`,
  `fields.intended_use`, and `fields.format`; evidence and proof descriptors
  also carry `fields.confidence` and `fields.provenance`.
- File-backed artifact descriptors give every file a `path` and `role`.
- Artifact descriptors have one purpose; files with different purposes are split
  into separate descriptors.
- Referenced non-`.ready` payload files have an artifact descriptor unless the
  reference is only a direct source/path link.
- Proof corpus descriptors use `type: proof_corpus` and carry
  `fields.proof_role`, `fields.authority_level`, `fields.dataset_contract`,
  `fields.oracle_policy`, `fields.replay`, `fields.acceptance_contract`, and
  `fields.limitations`.
- Sensitive content, source excerpts, diffs, raw logs, and secrets are absent.
- Large artifacts use storage paths or external refs instead of inline payloads.
- Settings files do not contain raw secrets, provider tokens, runtime cache, or
  product truth that belongs in primitives, governance, artifacts, or flags.

## Warning checks

- Primitive record body is too large to review comfortably.
- Premise confidence is low without a discovery flag or decision flag.
- Intent lacks failure behavior.
- Intent lacks claim level, process decision, or validation boundary.
- Intent lacks input/output sample refs when sample artifacts exist.
- Intent names a product surface only in prose instead of `required_surfaces`.
- Intent lacks required service or standard refs when its work depends on them.
- Standard governance uses a `child` ref where a `peer` ref would represent the
  same lateral governance more clearly. Child standard refs are still valid
  fallback semantics.
- Quality-bearing intent lacks an Operating Envelope.
- Service `status` is blocked without a blocker, evidence, proof required, or
  off-limits explanation.
- Product-shaping ambiguity appears only in prose.
- A canonical Ready branch contains speculative concept fragments that should be
  evaluated as an alternate Ready branch or worktree.
- A local semantic relationship display extension defines a source-to-target
  phrase without the matching target-to-source phrase.
- A generated or displayed target-state handoff is not traceable to product
  primitives, intent Completion Proof, relevant flags, standards, services, and
  artifacts.

## Importer behavior

Validators should report exact file paths and ids. They should not silently fix
semantic errors, average conflicting docs, treat evidence/artifact primitives as
approved product behavior, promote open flags into coding workflow, or turn
derived discrepancies into stored flags unless durable attention, blockers,
Completion Proof, claim coordination, or review history are needed.

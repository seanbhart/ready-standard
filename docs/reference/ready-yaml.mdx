# `.ready.yml` Reference

`.ready.yml` is the product-logic primitive source format. It is YAML with a
Ready-specific extension so tools can distinguish source records from navigation
Markdown, generated docs, manifests, and generic payload files.

Ready has a broader primitive model than this one schema. Flags, Decision flags,
artifact descriptors, and governance records are also Ready primitives when they are
typed, addressable, reusable records. They may use specialized schemas such as
`readyroom/flag/v1` or `readyroom/artifact/v1` when those schemas carry a more
precise machine contract.

## ID families

Product-logic ids keep their existing product prefixes, such as `P-*`, `I-*`,
`ST-*`, and `SV-*`.

All artifacts start with `A`. They may optionally use one or two additional
letters after `A` for a well-understood artifact class. For example, projects
may use `AR-*` for resources, `AP-*` for proof corpora, `ASN-*` for snippets,
`AD-*` for designs, and `AM-*` for manifests. Arbitrary or one-off artifacts
may use plain `A-*`.

All flags start with `X`. They may optionally use one or two additional letters
after `X` for a well-understood flag class. For example, projects may use
`XD-*` for decisions, `XS-*` for seed flags, `XC-*` for change flags, and
`XR-*` for resource/readiness flags. Arbitrary or one-off process interrupts
may use plain `X-*`.

## Product-Logic Primitive Record

```yaml
schema: readyroom/primitive/v1
kind: primitive
id: I-001
type: intent
primitive_class: product_logic
compiler_role: source_truth
title: Human-readable title
status: active
fields:
  statement: The concise product-promise statement.
  intent_type: behavior
  claim_level: process_proof
  actor: The user, operator, system, or agent this promise serves.
  trigger: The event or state that starts the promise.
  action: The behavior the actor or system performs.
  expected_end_state: The observable fulfilled state.
  value: Why the product promise matters.
  scope: What is included.
  non_scope: What is excluded.
  failure_behavior: What happens when the intent cannot work.
  process_decision: Current process choice or proof baseline.
  validation_boundary: What this intent does and does not validate.
  input_types: What the intent reads.
  input_samples: Sample artifact ids for representative inputs.
  output_types: What the intent produces.
  output_samples: Sample artifact ids for representative outputs.
  required_surfaces: User, operator, or internal surfaces required.
  operating_envelope: Quality/runtime limits when behavior can vary.
completion_proof:
  - Observable evidence that proves the intent exists.
refs:
  - role: parent
    id: P-001
artifacts: []
owner_notes: []
```

Required fields:

- `schema`: `readyroom/primitive/v1`
- `kind`: `primitive`
- `id`: stable primitive id, such as `P-001`, `I-001`, `ST-001`, or `SV-001`
- `type`: `premise`, `intent`, `standard`, or `service`
- `title`: human-readable label
- `status`: product status, such as `active`, `draft`, or `blocked`
- `fields`: type-specific structured content
- `completion_proof`: required for committed intents; recommended for any
  product-logic primitive that needs proof
- `refs`: directed structural relationships

Product-logic primitive subtypes live under `fields`, not top-level `type`.
Valid subtype values live in [`vocabulary.yaml`](../../vocabulary.yaml).
Constraints are `standard` records, not premise subtypes.

Recommended fields:

- `primitive_class`: `product_logic`
- `compiler_role`: one of `source_truth`, `constraint`, `dependency`,
  `evidence`, `workflow_state`, `generated_view_input`, or another documented
  project role

`owner_notes` is an optional product-owner annotation area. It exists for
human notes that should travel with the record but are not product truth,
requirements, evidence, Completion Proof, or agent instructions. Agents and
compilers must not satisfy, enforce, or cite `owner_notes` as source authority;
promote anything authoritative into typed `fields`, `refs`, `artifacts`, or
flags.

Older v1 records can infer `primitive_class: product_logic` from
`schema: readyroom/primitive/v1` and the four product-logic types.

Directory ownership is not stored inside primitive records. Tools derive record
authority from the file location declared by `ready/manifest.yaml`, usually
`ready/product/**` for product truth and `ready/flags/**` for workflow records.
Extra top-level keys may be ignored by tools.

## Root tree manifest

The root `ready/manifest.yaml` controls product, flags, governance, settings,
and local standard package locations. Use explicit manifest fields, not
directory sort order, to describe tree meaning.

```yaml
schema: readyroom/product-tree-root/v1
product: example
source_root: ready
product_directory: ready/product
flags_directory: ready/flags
governance_directory: ready/governance
settings_directory: ready/settings
ready_standard:
  version: "0.3.27"
  repository: "https://github.com/seanbhart/ready-standard"
  package_path: "ready/standard"
  package_manifest: "ready/standard/manifest.yaml"
```

`settings_directory` is optional but recommended when an app manages the Ready
tree. Files under `ready/settings/` are git-tracked workspace/app settings, not
primitive source records. They should use explicit settings schemas, avoid raw
secrets, and not duplicate product truth from primitives, governance,
artifacts, or flags.

Product primitives live under the configured `product_directory`. Flags and
other temporary workflow records live under the configured `flags_directory`.
Git commits, branches, and worktrees carry product evolution and alternate
concepts; the standard does not define additional product-state overlays.

## Primitive Relationships

Primitive refs store structural position only. Do not store semantic relationship
verbs such as `serves`, `requires`, `governed_by`, `contains_premise`, or
`questions` in source records. Those words are reader labels derived from the
source primitive type, target primitive type, and structural role.

Approved primitive ref roles:

| Role | Optional code | Meaning for compact refs |
| --- | ---: | --- |
| `peer` | `0` | Target is lateral to the current primitive. |
| `parent` | `1` | Target is a parent of the current primitive. |
| `child` | `2` | Target is a child of the current primitive. |

The numeric codes are an implementation convention for applications that want a
compact internal enum. Source YAML should prefer the readable role strings.

A ref may connect any primitive type to any other primitive type when the
structural relationship is true. Validators should validate role names and
target ids. They may warn on type pairs that are not listed in the display
lookup table, but the stored relationship remains only `peer`, `parent`, or
`child`.

Store one directed edge for each relationship. This reduces merge conflicts and
prevents source and inverse refs from drifting; views derive both forward and
reverse labels from the same stored edge.

The semantic display lookup lives in `vocabulary.yaml`. The source-to-target
phrase renders from the record that stores or owns the ref. The
target-to-source phrase renders when a view shows the same edge from the target
side. For example, `intent --parent--> premise` renders as the intent `serves`
the premise, and as the premise is `served by` the intent. Projects may extend
the display lookup table for local primitive classes, but every extension should
define both directions and must not add new stored relationship roles when a
display phrase will do.

Standards that govern premises, intents, services, flags, or other standards
usually use `peer` refs because the standard is lateral governance, not a
decomposed child. Child standard refs remain valid fallback semantics for older
trees or intentionally nested subordinate standards.

Reader tree nesting is a projection of the graph. A compact ref with
`role: parent` means the target should be treated as the current primitive's
parent for relationship display; a compact ref with `role: child` means the
target should be treated as the current primitive's child. Views may render
incoming refs by inverting those two roles.

Compact refs infer `from` as the current primitive id:

```yaml
refs:
  - role: child
    id: SV-001
  - role: peer
    id: ST-001
```

Use compact refs for relationships owned by the current primitive. If a record
must store an edge whose source is another primitive, include explicit `from`
and `to` so the direction remains unambiguous.

## Flag metadata

Flag records use `schema: readyroom/flag/v1` and keep the claim gate at the top
level. They point at intended product truth that needs attention, proof, or
coordination; they do not store implementation deltas.

```yaml
schema: readyroom/flag/v1
kind: flag
id: X-001
type: seed
primitive_class: decision_workflow
compiler_role: workflow_state
title: Build the first usable onboarding path
status: ready
claimable: true
blocked_by: []
completion_proof:
  - User can complete onboarding in a clean local run.
  - The documented verification command passes and the evidence artifact is linked.
fields:
  objective: What should be resolved or proven.
refs: []
artifacts: []
owner_notes: []
```

`completion_proof` is the Definition of Done for the flag. It must be concrete
proof content, not a one-word status such as `ready` or `done`. Coding handoffs
compile the resolved intended product state from product primitives, standards,
services, artifacts, and relevant flags before a coding agent claims work.

## Specialized Primitive Schemas

Decision/workflow and evidence/artifact primitives can use specialized schemas
instead of `readyroom/primitive/v1`. A Ready-defined `.ready.yml` record using
one of these schemas is a primitive. A raw JSON, HTML, image, CSV, or other
payload file under `artifacts/` is not itself a primitive unless it has a
Ready-defined schema/pattern; it is an attachment or safe ref owned by an
artifact primitive.

Examples:

- `readyroom/flag/v1`: decision/workflow primitive for attention, blockers,
  drift, proof gaps, claim coordination, and proof gates.
- `readyroom/artifact/v1`: evidence/artifact primitive for a typed artifact
  such as a customer statement, source document, reference resource, access
  reference, review protocol, proof corpus, code snippet, design artifact,
  or manifest. Artifact descriptors can support generated views and handoff
  material without making those support roles separate artifact types.
- `readyroom/governance/v1`: governance primitive for authority, process, skill,
  or template records when a project uses structured governance records.

Specialized primitive schemas still need stable ids, titles or labels,
provenance when relevant, privacy handling when relevant, and enough structured
metadata for search, graph traversal, validation, and compiled views. They do
not use a root `summary`; short explanatory text belongs in type-specific
`fields` only when it is real product data.

## Artifact descriptors

Artifact descriptors use artifact descriptor `.ready.yml` files. Typed manifests and
Markdown index rows can summarize or group artifacts, but they are
generated/navigation material, not the primitive source record. The descriptor
is the semantic Ready reference; raw JSON, JSONL, Markdown, HTML, image, CSV, or
other payload files are attachments or safe refs owned by a descriptor.

Every non-`.ready` file that is referenced by product, workflow, proof, or agent
work should have an artifact descriptor. Reference the descriptor id from
primitives, not the raw payload path, when the reference is meant to carry
meaning. Direct file/path links are still useful for source access, but they are
not semantic product references.

One descriptor records one artifact purpose. Do not use a descriptor as a folder
inventory or test bundle when its files have different purposes. A descriptor
may list multiple files only when those files are interchangeable versions or
representations of the same artifact purpose.

Required descriptor fields:

- `schema`
- `kind`
- `id`
- `type`
- `title`
- `status`
- `fields.purpose`
- `fields.content`
- `fields.intended_use`
- `fields.format`

Evidence and proof descriptors must also carry `fields.confidence` and
`fields.provenance`. File-backed descriptors must include `files[].path` and
`files[].role`.

Use `type` for the specific artifact subtype, not only a storage bucket. Common
types include `customer_statement`, `source_document`, `reference_resource`,
`access_reference`, `review_protocol`, `proof_corpus`, `code_snippet`,
`design_artifact`, and `manifest`. Put subtype data under `fields`; do not
duplicate it in sibling payload blocks with local-only ids.

```yaml
schema: readyroom/artifact/v1
kind: artifact
id: AP-010
type: proof_corpus
primitive_class: evidence_artifact
compiler_role: evidence
title: Synthetic replay proof run
status: active
privacy: sanitized_fixture
fields:
  confidence: medium
  provenance: Generated from local deterministic fixture data.
  purpose: Replay one deterministic proof run.
  content: Descriptor ids for the input corpus, scoring config, expected output, diagnostics, and replay contract.
  intended_use: Use as committed regression proof for the linked harness.
  format: yaml
  proof_role: regression
  authority_level: deterministic_fixture
  lifecycle_state: frozen
  coverage:
    proves:
      - deterministic replay shape
    does_not_prove:
      - live user quality
    forbidden_uses:
      - treating fixture success as production validation
  dataset_contract:
    input_sets:
      - AP-011
    config:
      - AP-012
    expected_outputs:
      - AP-013
    diagnostic_outputs:
      - AP-014
    generated_outputs: []
  oracle_policy:
    oracle_type: exact_match
    oracle_authority: committed_expected_outputs
    comparison_mode: inputs_plus_config_produce_expected_outputs
    ignored_fields: []
    invariants: []
    update_policy: owner_approved_fixture_update
  replay:
    verify_command: npm run verify:proof-corpus -- AP-010
    regenerate_command: npm run regenerate:proof-corpus -- AP-010
    regenerate_policy: owner_approval_required
  acceptance_contract:
    accepted_when:
      - verify command passes against committed expected outputs
    fails_when:
      - inputs plus config do not produce expected outputs
    evidence_required:
      - replay log
  limitations:
    - Fixture proof is not customer or production evidence.
artifacts:
  - AP-011
  - AP-012
  - AP-013
  - AP-014
usage_policy: []
owner_notes: []
```

Each non-interchangeable payload purpose gets its own descriptor:

```yaml
schema: readyroom/artifact/v1
kind: artifact
id: AP-011
type: proof_corpus
title: Synthetic replay input corpus
status: active
fields:
  confidence: medium
  provenance: Generated from local deterministic fixture data.
  purpose: Provide the replay input rows.
  content: Input rows consumed by the replay harness.
  intended_use: Use as the input artifact for AP-010.
  format: jsonl
files:
  - path: ready/product/artifacts/samples/synthetic-proof/inputs.jsonl
    role: primary
    purpose: Replay input rows.
```

Do not use artifact descriptors as reverse indexes for product primitives. Product
and workflow records point to artifacts through their own `artifacts` field.
Artifact descriptors may point to other artifacts through `artifacts` or `refs` when
the artifact depends on, derives from, includes, or carries another artifact.
When a proof run uses several non-interchangeable inputs, configs, expected
outputs, and diagnostics, give each purpose its own descriptor and have the run
or harness descriptor reference them by id.

```yaml
schema: readyroom/artifact/v1
kind: artifact
id: AR-011
type: customer_statement
primitive_class: evidence_artifact
compiler_role: evidence
title: Customer statement - buyer-specific pitch proof
status: active
privacy: user_provided_customer_statement
retrieved_at: '2026-06-19'
fields:
  confidence: high
  provenance: User-provided customer statement, 2026-06-19.
  purpose: Preserve one customer statement as direct evidence.
  content: A verbatim customer statement about buyer-specific pitch proof.
  intended_use: Use as evidence for premises or decisions that cite buyer-specific pitch needs.
  format: text
  statement: "I need to pitch my product in a way that shows how their existing customers will want to buy them."
  speaker_type: customer
  supports:
    - pitch strength
    - buyer-specific relevance
usage_policy: []
owner_notes: []
```

The premise that uses this statement owns the artifact link:

```yaml
artifacts:
  - AR-011
```

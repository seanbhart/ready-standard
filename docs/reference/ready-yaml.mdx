# `.ready.yml` Reference

`.ready.yml` is the product-logic primitive source format. It is YAML with a
Ready-specific extension so tools can distinguish source records from navigation
Markdown, generated docs, manifests, and generic payload files.

Ready has a broader primitive model than this one schema. Flags, questions,
artifact records, and governance records are also Ready primitives when they are
typed, addressable, reusable records. They may use specialized schemas such as
`readyroom/flag/v1` or `readyroom/artifact/v1` when those schemas carry a more
precise machine contract.

## Product-Logic Primitive Record

```yaml
schema: readyroom/primitive/v1
kind: primitive
id: I-001
type: intent
title: Human-readable title
milestone: m1
status: active
fields:
  statement: The primary product-truth statement.
  value: Why the product promise matters.
  input_data: What the intent reads.
  output_data: What the intent produces.
refs:
  - from: I-001
    role: serves
    to: P-001
artifacts: []
owner_notes: []
```

Required fields:

- `schema`: `readyroom/primitive/v1`
- `kind`: `primitive`
- `id`: stable primitive id, such as `P-001`, `I-001`, `ST-001`, or `SV-001`
- `type`: `premise`, `intent`, `standard`, or `service`
- `title`: human-readable label
- `milestone`: owning stage id; the v1 field name remains `milestone` even
  though the root manifest standardizes the grouping as `stages`
- `status`: product status, such as `active`, `draft`, or `blocked`
- `fields`: type-specific structured content
- `refs`: directed semantic edges

Recommended fields:

- `primitive_class`: `product_logic`
- `compiler_role`: one of `source_truth`, `constraint`, `dependency`,
  `evidence`, `workflow_state`, `generated_view_input`, or another documented
  project role

`owner_notes` is an optional product-owner annotation area. It exists for
human notes that should travel with the record but are not product truth,
requirements, evidence, Completion Proof, or agent instructions. Agents and
compilers must not satisfy, enforce, or cite `owner_notes` as source authority;
promote anything authoritative into typed `fields`, `refs`, `artifacts`, or
flags.

Older v1 records can infer `primitive_class: product_logic` from
`schema: readyroom/primitive/v1` and the four product-logic types.

## Root stage registry

The root `ready/manifest.yaml` controls stage grouping and default selection.
Use `stages`, not directory sort order, to describe stage meaning. Stage ids
may be `m1`, `m2`, `alpha`, `client-pilot`, or another project-specific id;
the examples currently use `m1`.

```yaml
schema: readyroom/product-tree-root/v1
product: example
source_root: ready
default_stage: m1
stages:
  - id: m1
    title: M1
    path: ready/m1
    manifest: ready/m1/manifest.yaml
    kind: normal
    status: active
    order: 10
    default_candidate: true
    coding_claims_enabled: false
  - id: end-state
    title: End State
    path: ready/end-state
    manifest: ready/end-state/manifest.yaml
    kind: horizon
    status: horizon
    order: 9999
    default_candidate: false
    coding_claims_enabled: false
```

If `default_stage` is absent, tools choose the latest normal default candidate
by `order`, then the latest normal stage, and only then non-normal stages.

## Ref roles

A ref may connect any primitive type to any other primitive type when the role
semantics are true. Validators should validate role names and target ids, not a
fixed product-logic type-pair matrix.

Approved primitive ref roles and semantics:

- `serves`: source supports, enables, addresses, or is intentionally in service
  of target. Services and standards may use `serves` to show they serve any
  primitive type.
- `contains_premise`: source owns a subordinate premise that should appear
  under the source instead of as a root premise.
- `requires`: source cannot be satisfied, proven, or used without target.
- `governed_by`: source is constrained by target standard or governance
  primitive.
- `questions`: source raises an unresolved question about target.

Store one directed edge for each relationship. This reduces merge conflicts and
prevents source and inverse refs from drifting; views can derive inverse labels.

Ref storage ownership follows dependency and implementation ownership, not the
visual tree direction. Store the edge on the record whose readiness,
completeness, implementation, or interpretation depends on the relationship.
Use these tie-breakers:

- Flags always own their relationships, including relationships to intents,
  services, standards, premises, and artifacts.
- Artifact records do not own relationships to product or workflow primitives.
  Product/workflow records point to artifacts through `artifacts`. Artifact
  records may own artifact-to-artifact relationships when one artifact depends
  on, derives from, includes, or carries another artifact.
- If an intent participates in a non-flag relationship, store the relationship
  on the intent.
- If no intent participates but a service does, store the relationship on the
  service.
- Same-type relationships follow dependency direction: the record that depends
  on, derives from, contains, blocks, questions, or requires the other owns the
  edge.

Common product-tree edges:

- `intent --serves--> premise`: the intent addresses a root product premise.
- `intent --contains_premise--> premise`: the intent owns a subordinate premise
  that should appear under that intent instead of as a root premise.
- `intent --requires--> service`: the intent depends on a service.
- `intent --governed_by--> standard`: the intent must obey a standard.
- `service --serves--> <any primitive>`: the service exists to support that
  primitive.
- `standard --serves--> <any primitive>`: the standard exists to support that
  primitive.

Do not store an inverse ref for the same assertion. For example, choose either
`intent --requires--> service` or `service --serves--> intent` for the
relationship you mean; reader views derive inverse labels automatically.

Reader tree nesting is a projection of the graph, not the storage direction.
Common projections include premises as roots, intents under the premises they
serve, services and standards under the intents that require or are governed by
them, and contained premises under the owning intent. Reader views may render
incoming refs with inverse labels such as `Served by` or `Required by`.

Compact refs infer `from` as the current primitive id:

```yaml
refs:
  - role: requires
    id: SV-001
```

Use compact refs only for outgoing edges from the current primitive. If a record
stores an edge whose source is another primitive, include explicit `from` and
`to` so the canonical direction remains unambiguous.

## Flag metadata

Flag records use `schema: readyroom/flag/v1` and keep the claim gate at the top
level:

```yaml
schema: readyroom/flag/v1
kind: flag
id: F-001
type: seed
primitive_class: decision_workflow
compiler_role: workflow_state
title: Build the first usable onboarding path
milestone: m1
status: ready
claimable: true
blocked_by: []
completion_proof:
  - User can complete onboarding in a clean local run.
  - The documented verification command passes and the evidence artifact is linked.
fields:
  objective: What should be resolved or proven.
refs: []
artifacts: []
owner_notes: []
```

`completion_proof` is the Definition of Done for the flag. It must be concrete
proof content, not a one-word status such as `ready` or `done`.

## Specialized Primitive Schemas

Decision/workflow and evidence/artifact primitives can use specialized schemas
instead of `readyroom/primitive/v1`. A Ready-defined `.ready.yml` record using
one of these schemas is a primitive. A raw JSON, HTML, image, CSV, or other
payload file under `artifacts/` is not itself a primitive unless it has a
Ready-defined schema/pattern; it is an attachment or safe ref owned by an
artifact primitive.

Examples:

- `readyroom/flag/v1`: decision/workflow primitive for attention, blockers,
  drift, proof gaps, seed work, or delta work.
- `readyroom/artifact/v1`: evidence/artifact primitive for a typed artifact
  such as a customer statement, source document, reference resource, access
  reference, review protocol, sample data, code snippet, design artifact,
  manifest, generated view, or handoff record.
- `readyroom/governance/v1`: governance primitive for authority, process, skill,
  or template records when a project uses structured governance records.

Specialized primitive schemas still need stable ids, titles or labels,
provenance when relevant, privacy handling when relevant, and enough structured
metadata for search, graph traversal, validation, and compiled views. They do
not use a root `summary`; short explanatory text belongs in type-specific
`fields` only when it is real product data.

## Artifact metadata

Artifact records use artifact `.ready.yml` files. Typed manifests and Markdown
index rows can summarize or group artifacts, but they are generated/navigation
material, not the primitive source record. At minimum, an artifact primitive
should identify the id, specific `type`, privacy state, confidence, proof role,
and any attached file path or safe ref.

Use `type` for the specific artifact subtype, not only a storage bucket. Common
types include `customer_statement`, `source_document`, `reference_resource`,
`access_reference`, `review_protocol`, `sample_data`, `code_snippet`,
`design_artifact`, and `manifest`. Put subtype data under `fields`; do not
duplicate it in sibling payload blocks with local-only ids.

```yaml
schema: readyroom/artifact/v1
kind: artifact
id: SA-001
type: sample_data
primitive_class: evidence_artifact
compiler_role: evidence
title: Synthetic fixture
milestone: m1
status: active
privacy: sanitized_fixture
fields:
  confidence: medium
  fixture_scope: Synthetic input fixture.
files:
  - path: ready/m1/artifacts/samples/payloads/SA-001.json
    purpose: Synthetic input fixture.
usage_policy: []
owner_notes: []
```

Do not use artifact records as reverse indexes for product primitives. Product
and workflow records point to artifacts through their own `artifacts` field.
Artifact records may point to other artifacts through `artifacts` or `refs` when
the artifact depends on, derives from, includes, or carries another artifact.

```yaml
schema: readyroom/artifact/v1
kind: artifact
id: RE-011
type: customer_statement
primitive_class: evidence_artifact
compiler_role: evidence
title: Customer statement - buyer-specific pitch proof
milestone: m1
status: active
privacy: user_provided_customer_statement
retrieved_at: '2026-06-19'
fields:
  confidence: high
  statement: "I need to pitch my product in a way that shows how their existing customers will want to buy them."
  speaker_type: customer
  supports:
    - pitch strength
    - buyer-specific relevance
usage_policy: []
owner_notes: []
```

The premise that uses this statement owns the artifact link:

```yaml
artifacts:
  - RE-011
```

# `.ready` Source Records

The Ready Standard defines `.ready` source records. This version stores those
records as `.ready.yml`: YAML with a distinct extension so tools can separate
typed source records from navigation Markdown, generated views, manifests, raw
payloads, and generic project files.

Use `.ready.yml` for source records until the standard promotes `.ready` as the
canonical file extension. Treat `.ready` as the durable format identity and
`.ready.yml` as the current YAML serialization.

`.ready` records use three classification fields:

- `kind`: the object family, such as `primitive`, `flag`, `governance`,
  `setting`, or `manifest`.
- `class`: the family-specific class. Primitive classes are `premise`,
  `intent`, `service`, `standard`, and `artifact`.
- `type`: the narrower subtype under that class.

The valid values live in [`vocabulary.yaml`](../../vocabulary.yaml). This is a
breaking contract: do not write `primitive_class`, `fields.premise_type`, or
`fields.intent_type` in new records.

## ID Families

Product primitive ids use the class prefix:

- `P-*` for premises.
- `I-*` for intents.
- `ST-*` for standards.
- `SV-*` for services.
- `A*` for artifact primitives.

Artifacts may optionally use one or two additional letters after `A` for a
well-understood artifact type, such as `AR-*` for resources, `AP-*` for proof
corpora, `ASN-*` for snippets, `AD-*` for designs, and `AM-*` for manifests.
Plain `A-*` is valid for one-off artifacts.

Flags use `X*` ids. They may optionally use one or two letters after `X` for a
well-understood flag class, such as `XD-*` for decisions, `XB-*` for blockers,
`XDR-*` for discrepancies, `XR-*` for readiness, and `XP-*` for proof gaps.

## Primitive Records

Primitive records are durable Ready package inputs required to build, prove,
operate, or govern the intended product package. They are the source of intended
product truth, implementation constraints, dependencies, proof inputs, or
supporting product artifacts. Most primitives live under the configured
`product_directory`, usually `ready/product/`.

```yaml
schema: ready/primitive/v1
kind: primitive
id: I-001
class: intent
type: behavior
title: Human-readable title
status: ready
fields:
  statement: The concise product-promise statement.
  claim_level: process_proof
  actor: The user, operator, system, or agent this promise serves.
  trigger: The event or state that starts the promise.
  action: The behavior the actor or system performs.
  expected_end_state: The observable fulfilled state.
  value: Why the product promise matters.
  scope: What is included.
  non_scope: What is excluded.
  failure_behavior: What happens when the intent cannot work.
  process_decision: Current process choice or proof baseline.
  validation_boundary: What this intent does and does not validate.
  input_types: What the intent reads.
  input_samples: Sample artifact ids for representative inputs.
  output_types: What the intent produces.
  output_samples: Sample artifact ids for representative outputs.
  required_surfaces: User, operator, or internal surfaces required.
  operating_envelope: Quality/runtime limits when behavior can vary.
completion_proof:
  - Observable evidence that proves the intent exists.
refs:
  - role: parent
    id: P-001
artifacts: []
owner_notes: []
```

Required primitive fields:

- `schema`: usually `ready/primitive/v1`; specialized primitive schemas may
  be used when they preserve a narrower machine contract.
- `kind`: `primitive`.
- `id`: stable primitive id.
- `class`: `premise`, `intent`, `standard`, `service`, or `artifact`.
- `type`: a valid subtype for the class.
- `compiler_role`: how compilers and orchestrators should treat the record.
- `title`: human-readable label.
- `status`: record state: `ready`, `draft`, or `blocked`.
- `fields`: class-specific structured content.
- `refs`: directed structural relationships.

Committed intents should carry `completion_proof`. Other primitives should add
Completion Proof when the record needs explicit proof to rebuild, validate, or
operate the product.

`owner_notes` is an optional product-owner annotation area. It exists for human
notes that should travel with the record but are not product truth,
requirements, evidence, Completion Proof, or agent instructions. Agents and
compilers must not satisfy, enforce, or cite `owner_notes` as source authority;
promote anything authoritative into typed `fields`, `refs`, `artifacts`, or
flags.

Directory ownership is not stored inside primitive records. Tools derive record
authority from the file location declared by `ready/manifest.yaml`.

## Status

Ready records use top-level `status` to communicate record state and
kind-aware orchestrator behavior.

- `ready`: resolved for its record kind. Product primitives compile as current
  intended truth. Flags do not use `ready` status.
- `draft`: incomplete or not yet approved for its record kind. An orchestrator
  may attempt implementation, but it must warn before doing so and the build
  should not be expected to produce a complete product.
- `blocked`: cannot be built, proven, operated, or claimed without resolving
  the blocker or receiving explicit override instructions.

Use these statuses consistently wherever a Ready source record carries
top-level `status`.

## Root Tree Manifest

The root `ready/manifest.yaml` controls product, flags, governance, settings,
and local standard package locations. Use explicit manifest fields, not
directory sort order, to describe tree meaning.

Manifest paths are repo-root relative unless a manifest declares a different
`path_base`.

```yaml
schema: ready/product-tree-root/v1
kind: manifest
class: product
product: example
source_root: ready
product_directory: ready/product
flags_directory: ready/flags
governance_directory: ready/governance
settings_directory: ready/settings
ready_standard:
  version: "0.3.32"
  repository: "https://github.com/seanbhart/ready-standard"
  package_path: "ready/standard"
  package_manifest: "ready/standard/manifest.yaml"
```

`settings_directory` is optional but recommended when an app manages the Ready
tree. Files under `ready/settings/` are git-tracked workspace/app settings, not
primitive source records. They should use explicit settings schemas, avoid raw
secrets, and not duplicate product truth from primitives, governance,
artifacts, or flags.

Product primitives live under the configured `product_directory`. Flags and
other temporary package-work records live under the configured
`flags_directory`.
Git commits, branches, and worktrees carry product evolution and alternate
concepts; the standard does not define additional product-state overlays.

## Relationships

Refs store structural position only. Do not store semantic
relationship verbs such as `serves`, `requires`, `governed by`, `refines`, or
`interrupts` in source records. Those words are reader labels derived from the
source kind and class, target kind and class, and structural role.

Approved ref roles:

| Role | Optional code | Meaning for compact refs |
| --- | ---: | --- |
| `peer` | `0` | Target is lateral to the current record. |
| `parent` | `1` | Target is a parent of the current record. |
| `child` | `2` | Target is a child of the current record. |

The numeric codes are an implementation convention for applications that want a
compact internal enum. Source YAML should prefer the readable role strings.

A ref may connect records across Ready kinds when the structural relationship
is true. Validators should validate role names, target ids, and whether the
kind/class pairing is allowed by project policy. They may warn on pairs that
are not listed in the display lookup table, but the stored relationship remains
only `peer`, `parent`, or `child`.

Store one directed edge for each relationship. This reduces merge conflicts and
prevents source and inverse refs from drifting; views derive both forward and
reverse labels from the same stored edge.

The semantic display lookup lives in `vocabulary.yaml`. The source-to-target
phrase renders from the record that stores or owns the ref. The
target-to-source phrase renders when a view shows the same edge from the target
side. For example, `intent --parent--> premise` renders as the intent `serves`
the premise, and as the premise is `served by` the intent. A
`flag --peer--> standard` edge can render as `governed by` / `governs` without
making the flag a primitive.

Standards govern implementation, behavior, measurement, judgment, or
maintenance. Attach standards to intents, services, flags, or other standards,
not to premises. Services are implementation dependencies; attach services to
intents or other implementation-bearing records, not to premises. If a service
or standard appears to relate to a premise, attach it to the intent that serves
the premise.

## Flags

Flags are not primitives. They are temporary package-work records for unresolved
Ready-package decisions, proposed changes, version divergence, blockers,
discrepancies, readiness gates, and proof gaps. A complete, stable Ready package
has no flags.

```yaml
schema: ready/flag/v1
kind: flag
id: XR-001
class: readiness
type: seed
title: Build the first usable onboarding path
claimable: true
blocked_by: []
completion_proof:
  - User can complete onboarding in a clean local run.
  - The documented verification command passes and the evidence artifact is linked.
fields:
  objective: What should be resolved or proven.
refs: []
artifacts: []
owner_notes: []
```

`completion_proof` is closure criteria for the flag. It must be concrete proof
content, not a one-word status such as `ready` or `done`. Coding handoffs
compile the resolved intended product state from primitives, standards,
services, artifacts, and relevant flag closure criteria before a coding agent
claims work.

Decision flags use `class: decision` and a decision subtype such as
`type: product_choice`. Keep the unresolved prompt in `fields.question`, the
conservative interim behavior in `fields.safe_default`, candidate answers in
structured `fields.proposals`, and the decision closure criteria in top-level
`completion_proof`. Nested proposal entries should use
`recommendation_state`, not `status`; top-level `status` is reserved for Ready
record state.

## Artifact Primitives

Artifact descriptors are primitives with `class: artifact`. They describe one
durable supporting artifact purpose with stable ids, provenance, privacy state,
usage roles, and payload files or safe refs. Raw JSON, JSONL, Markdown, HTML,
image, CSV, binary, or external payloads are attachments owned by an artifact
primitive; they are not primitives by themselves unless they also use a
Ready-defined source-record schema.

Artifact descriptors may use `schema: ready/artifact/v1` when that narrower
schema gives tools a clearer contract. The object model is still:

```yaml
schema: ready/artifact/v1
kind: primitive
id: AP-010
class: artifact
type: proof_corpus
title: Synthetic replay proof run
status: ready
privacy: sanitized_fixture
fields:
  confidence: medium
  provenance: Generated from local deterministic fixture data.
  purpose: Replay one deterministic proof run.
  content: Descriptor ids for the input corpus, scoring config, expected output, diagnostics, and replay contract.
  intended_use: Use as committed regression proof for the linked harness.
  format: yaml
  proof_role: regression
  authority_level: deterministic_fixture
  lifecycle_state: frozen
  coverage:
    proves:
      - deterministic replay shape
    does_not_prove:
      - live user quality
    forbidden_uses:
      - treating fixture success as production validation
  dataset_contract:
    input_sets:
      - AP-011
    config:
      - AP-012
    expected_outputs:
      - AP-013
    diagnostic_outputs:
      - AP-014
    generated_outputs: []
  oracle_policy:
    oracle_type: exact_match
    oracle_authority: committed_expected_outputs
    comparison_mode: inputs_plus_config_produce_expected_outputs
    ignored_fields: []
    invariants: []
    update_policy: owner_approved_fixture_update
  replay:
    verify_command: npm run verify:proof-corpus -- AP-010
    regenerate_command: npm run regenerate:proof-corpus -- AP-010
    regenerate_policy: owner_approval_required
  acceptance_contract:
    accepted_when:
      - verify command passes against committed expected outputs
    fails_when:
      - inputs plus config do not produce expected outputs
    evidence_required:
      - replay log
  limitations:
    - Fixture proof is not customer or production evidence.
artifacts:
  - AP-011
  - AP-012
  - AP-013
  - AP-014
usage_policy: []
owner_notes: []
```

Every non-`.ready` file that supports product logic, flags, proof, generated
views, or handoff should have an artifact primitive. Governance documents and
vendored Ready standard package resources keep their own record kinds and do not
need artifact descriptors. Reference the artifact id from primitives and flags
when the reference is meant to carry semantic meaning. Direct file/path links
are still useful for source access, but they are not semantic product
references.

One descriptor records one artifact purpose. Do not use a descriptor as a
folder inventory or test bundle when its files have different purposes. A
descriptor may list multiple files only when those files are interchangeable
versions or representations of the same artifact purpose.

Required artifact fields:

- `schema`
- `kind`
- `id`
- `class`
- `type`
- `compiler_role`
- `title`
- `status`
- `fields.purpose`
- `fields.content`
- `fields.intended_use`
- `fields.format`

Artifact descriptors must also carry `fields.confidence` and
`fields.provenance`. File-backed descriptors must include `files[].path` and
`files[].role`.

Artifact-tree manifests may declare `artifact_type_directory_base:
"artifact_root"` to make `artifact_types.*.directory` values relative to
`artifact_root`.

`type: manifest` under `class: artifact` means the artifact primitive describes
a durable product artifact whose purpose is a manifest or index. This is
different from `kind: manifest`, which is structural Ready package metadata.

## Governance, Settings, And Manifests

Governance records are not primitives by default. They control who may change
the Ready package, how product truth is approved, and how agents should behave;
they are package management records, not build inputs for the product runtime.
Use `kind: governance` for structured governance source records.

Settings records are not primitives. Use `kind: setting` for git-tracked
workspace or app configuration that tools need to manage the Ready tree. Do not
store raw secrets, runtime cache, or product truth in settings.

Manifest records are structural metadata. Use `kind: manifest` for product tree
manifests, package manifests, artifact-tree manifests, governance manifests, and
settings manifests.

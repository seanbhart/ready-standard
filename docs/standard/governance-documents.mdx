# Governance And Standard Resources

Governance documents are governance records: package-management records that
constrain authority, process, and agent behavior. They are not primitives or
artifact primitives by default. Ready Standard resources are reusable package
resources; their own manifests, templates, and skills keep their native record
kind rather than automatically becoming governance records.

Use `ready/governance/` for project-specific files that constrain how product
work is allowed to happen for that project:

- Agent behavior guidelines.
- Product-orchestrator charters.
- Product process docs.
- Workspace authority and access policies.

These files travel with the Ready package so every contributor and agent uses
the same authority and process versions. They are package-management context,
not product runtime implementation truth, but they can stop or constrain package
edits and implementation handoffs.

Use the Ready Standard package for project-agnostic resources:

- Ready Skill packages.
- Product task-skill modules.
- Governance templates.
- Manifest templates.
- Primitive templates.

Product repos that expect agents to edit a Ready tree should keep a pinned copy
of those resources at `ready/standard/`. That local copy is the runtime standard
package for agents. The source repo's `docs/` directory is Mintlify
documentation source and is not part of the product-local package.

Use `ready/product/artifacts/` for durable product artifact primitives and
attached payload material used by product logic, proof, generated views, or
handoff. A flag may own workflow-only artifacts under `ready/flags/` when they
only support temporary package-work:

- Proof corpora.
- Code snippets.
- Design assets.
- Portal manifests.
- Screenshots.
- Handoff notes.

## Rule

If a file tells an agent what authority it has, what project-specific process it
must follow, or what project-specific standard wins during conflict, it belongs
in governance.

If a file defines reusable Ready procedure or a starter template for other
projects, it belongs in the standard package, not in a project artifact tree.

If a file helps explain, test, prove, or illustrate product logic, it belongs in
artifacts as an artifact primitive or as a payload referenced by one.

## Runtime use

Agent context manifests should record the governance documents and standard
package resources loaded for each run, including versions, source refs, and
hashes. This keeps agent behavior auditable without treating private chats,
local scratchpads, or generated coding handoffs as governance.

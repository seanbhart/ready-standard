# Primitive Records

Primitive records are typed, addressable Ready records. Product-logic
primitives use `.ready.yml` files with `schema: readyroom/primitive/v1`.
Decision/workflow primitives, evidence/artifact primitives, and governance
primitives may use specialized schemas such as `readyroom/flag/v1` or
`readyroom/artifact/v1` when those schemas are the honest contract for the
record.

Humans normally read compiled views in the Ready Room app or docs; the raw file
is optimized for parsing, validation, diffing, and merge review.

## Primitive classes

Use primitive class to describe what kind of authority a record carries:

- `product_logic`: premises, intents, standards, and services that define the
  product behavior or constraints.
- `decision_workflow`: question cards, flags, blockers, drift, proof gaps, and
  other workflow or ambiguity records.
- `evidence_artifact`: artifact records for resources, samples, snippets,
  designs, manifests, and handoff material. The record is the primitive; large
  or binary payload files are attachments or safe refs.
- `governance`: authority, process, agent, skill, and template records that
  constrain how product truth may change.

Compiler role matters as much as class. A sample-data primitive may be valid
evidence without being approved product behavior. A seed-flag primitive may be
workflow state without being implementation permission until its claim gate
passes.

## Common rules

Every product-logic primitive record should include:

```yaml
schema: readyroom/primitive/v1
kind: primitive
id: I-001
type: intent
title: Browse and edit the primitive-backed product spine
milestone: m1
status: active
summary: The concise product-truth statement.
fields:
  value: Why this matters to the user or operator.
  failure_behavior: What happens when the intent cannot be fulfilled.
refs:
  - from: I-001
    role: serves
    to: P-001
artifacts: []
owner_notes: []
```

Put every value an app, compiler, validator, search index, or agent must read in
structured fields. Reader-friendly Markdown, PM summaries, developer briefs,
designer views, and customer-facing docs are compiled from these records.

## File rules

- Use one primitive record per source file.
- Use the `.ready.yml` extension for product-logic primitive source records.
- Use specialized Ready schemas for decision/workflow, artifact, or governance
  primitives when a narrower schema preserves a clearer contract.
- Keep ids stable; paths and titles can change.
- Keep files fragmented and reviewable.
- Store relationships in `refs`, not prose.
- Any primitive type may relate to any other primitive type when the ref role
  semantics are true.
- Store one directed edge for each relationship so source and inverse refs do
  not drift during concurrent edits; compiled views derive inverse labels.
- Use approved ref roles with clear semantics: `serves` means source supports or
  enables target; `contains_premise` means source owns a subordinate premise;
  `requires` means source depends on target; `governed_by` means target
  constrains source; `questions` means source raises unresolved ambiguity about
  target.
- Product-tree examples include `intent --serves--> premise`,
  `intent --contains_premise--> premise`, `intent --requires--> service`, and
  `intent --governed_by--> standard`. Services and standards may also use
  `serves` when they serve any primitive type.
- Do not store an inverse ref for the same assertion. Choose the directed edge
  that states the fact you mean.
- Compact refs infer `from` as the current primitive id, so use compact refs
  only for outgoing edges from the current primitive.
- Store bulky samples, mockups, screenshots, snippets, and generated outputs as
  artifact payloads referenced by an evidence/artifact primitive id or path.
- Do not store local chat logs, raw provider transcripts, source diffs, secrets,
  or unreviewed scratch content in primitive records.
- Do not store coding lifecycle state on product-logic primitives. Seed and
  delta flags are decision/workflow primitives that gate coding claims.

## Premises

Premises explain why the product should exist. They need evidence confidence and
provenance in `fields`. A premise inferred only from docs should be marked as
inferred or low confidence until the product lead confirms it.

## Intents

Intents are product promises. Each committed intent should have an actor, action,
expected end state, value, scope, non-scope, failure behavior, input/output data,
required services, required standards, and an Operating Envelope when behavior
can vary materially after release.

Completion Proof does not live on persistent intent bodies. It belongs on ready
seed or delta flags.

## Standards

Standards state rules for implementation, measurement, judgment, behavior, or
maintenance. Global standards apply through workspace standard refs; primitive
specific standards apply through directed `governed_by` refs.

## Services

Services capture dependencies required to build, prove, run, or monitor the
product. A service should name readiness by environment, access expectations,
proof policy, failure modes, simulation rules, observability, privacy, owner,
and readiness gaps.

## Question Cards

Question cards are decision/workflow primitives. They hold product-shaping
ambiguity. When answered, apply the resulting primitive edits or flags and
remove the active question card. History belongs in Git, events, assessments,
or implementation refs.

## Flags

Flags are decision/workflow primitives. They mark attention, blockers, drift,
proof gaps, and coding-readiness state without putting lifecycle state into
product-logic primitive bodies. A flag can be a primitive without being a coding
ticket; claimability still depends on flag type, status, blockers, Completion
Proof, and workspace policy. Flag records keep `type`, `claimable`, and
`completion_proof` at the top level.

## Artifact Records

Artifact records are evidence/artifact primitives. They describe resources,
samples, snippets, designs, manifests, screenshots, generated views, or handoff
material with stable ids, provenance, privacy state, and usage roles. The
record is primitive source. The attached file may be JSON, Markdown, HTML,
image, binary, external URL, or safe secret-location reference.

# Flags and Claims

Flags are decision/workflow primitives. They are not product-logic primitives,
not tickets, and not implementation instructions. They are attention,
coordination, and proof-gate records attached to intended product truth.

The intended product state belongs in product-logic primitives. A flag should
not restate the whole target spec or prescribe how to change the implementation.
It should identify the unsatisfied or unproven intended state, name blockers or
claim policy, and define the observable evidence that closes the gap.

Every flag id starts with `X`. A flag may use a one- or two-letter class suffix
after `X` when the class is well understood: for example `XD-*` for decisions,
`XS-*` for seed flags, `XC-*` for change flags, or `XR-*` for
resource/readiness flags. Plain `X-*` is valid for arbitrary or one-off process
interrupts. An X primitive should close once the required process action or
decision is complete; durable results belong on the affected product primitive,
artifact, proof, workflow state, or trace.

## Flag types

- `discovery`: premise or product discovery before intent work is ready.
- `seed`: proof/attention gate for intended product truth that is not yet
  satisfied by implementation.
- `change`: proof/attention gate for intended product truth that changed and is
  not yet satisfied by implementation.
- `coverage_gap`: missing primitive coverage.
- `blocker`: claim, proof, service, authority, or environment blocker.
- `drift`: detected divergence between product truth, implementation, evidence,
  generated views, or prior Ready commits.
- `proof_gap`: missing proof detail.
- `decision`: product-shaping ambiguity that needs a Decision flag. Use the
  dedicated Decision template with an `XD-*` id, `fields.question`,
  `fields.safe_default`, structured `fields.proposals`, and top-level
  `completion_proof`.

## Claim gate

A coding agent can claim a flag only when all of these are true:

```yaml
type: seed # or change
status: ready
blocked_by: []
claimable: true
completion_proof:
  - "User can complete the target workflow in a clean local run."
  - "The relevant verification command passes and the evidence artifact is linked."
```

The workspace Coding work switch must also be enabled.

Claiming a flag does not mean applying a stored delta. The coding handoff should
give the coding agent the resolved intended product state from `ready/product/`,
plus standards, services, artifacts, intent Completion Proof, and any relevant
flag closure criteria. The coding agent may choose the clean implementation
strategy, including replacement or refactor, inside those product and proof
constraints.

## Open flags

An open discovery, seed, or change flag may be sparse. It only needs to point to
the intended product truth that needs attention and preserve the trigger context.
Open seed and change flags are product/design responsibility, not coding
workflow.

## Ready but blocked

A seed or change flag may be `ready` while still blocked, but it must be
non-claimable:

```yaml
status: ready
claimable: false
blocked_by:
  - SV-003
completion_proof:
  - "The target workflow is implemented and verified with the linked service stub."
```

This is useful when the intended product state and Completion Proof are well
specified, but a service, authority decision, environment, or upstream flag still
prevents implementation.

## Completion Proof

Intents carry the durable Completion Proof needed to rebuild the product from
the Ready tree. Ready seed and change flags may also carry top-level
`completion_proof` as closure criteria for the temporary flag: the observable
outcomes and evidence that remove the flag. It must be real proof content, not
a one-word status such as `ready`, `done`, or `complete`, and it does not
prescribe file edits.

## Derived discrepancies

Apps may derive unstored discrepancies by comparing the current Ready tree to
the codebase, proof artifacts, generated views, assessment snapshots, or prior
Ready commits. Store a flag only when the discrepancy needs durable human
attention, explicit Completion Proof, blocker tracking, claim coordination, or
review history. Otherwise the discrepancy can remain an interpreted app view.

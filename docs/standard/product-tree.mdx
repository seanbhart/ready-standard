# `.ready` Product Tree Standard

A `.ready` product tree is a committed filetree that stores product truth in
typed records. Primitive source records currently use `.ready.yml`: plain YAML
with a distinct extension so importers know exactly how to parse it. The
standard uses `.ready` as the durable bundle and record identity; `.ready.yml`
is the current YAML serialization. The tree should behave like a healthy
codebase: fragmented enough to review, explicit enough to validate, and
structured enough to compile into different views.

A `.ready` tree on a branch represents the intended product state for that
branch. It should say what should exist, not how to change the current
implementation. Git history records how product truth evolved; Ready primitives
describe the currently intended state.

A complete Ready bundle has enough ready product truth, standards, services,
artifacts, proof, governance, settings, and manifests for an orchestrator to
build from the bundle without relying on unstated tickets, chat memory, or the
current code structure. Incomplete bundles are still useful, but an
implementation attempt from an incomplete bundle should be labeled as such.

## Source of truth

Each primitive owns its durable truth in one `.ready.yml` file:

- The YAML document is the machine contract.
- Apps and docs compile reader-friendly views from the YAML.
- Stable ids are the durable identity.
- Paths and titles are refactorable.
- Generated views derive from the tree.

Markdown is still useful for navigation, governance, and generated views, but it
is not the primitive source format. Top-level index files are allowed for
navigation, but they must not duplicate record truth.

Hypothetical ideas and concepts should usually be modeled as alternate Ready
branches or worktrees rather than speculative fragments inside the canonical
branch. A decision or flag can point to that branch when the team needs to choose
or coordinate, but the idea's product shape should remain a coherent intended
state tree.

## Tree directories

The root `ready/manifest.yaml` owns the product, flags, governance, settings,
and standard bundle locations. The standard does not define additional
product-state overlays. A Ready tree version should state one coherent intended product; Git
commits, branches, and worktrees carry history, alternatives, and hypothetical
concepts.

Use `product_directory`, `flags_directory`, `governance_directory`, and
`settings_directory` in the root manifest:

```yaml
product_directory: ready/product
flags_directory: ready/flags
governance_directory: ready/governance
settings_directory: ready/settings
```

`ready/product/` is the canonical source of intended product truth for the
current tree version. It holds premises, intents, standards, services, product
artifacts, proof corpora, and other records that define what should exist.

`ready/flags/` is a peer workflow directory. It may hold temporary bundle-work
records for unresolved Ready-bundle decisions, proposed changes,
version-divergence notes, blockers, discrepancies, readiness gates, proof gaps,
and claim coordination. Flags can point at product truth, but they do not own
the product truth and they do not replace intent Completion Proof. A complete
stable bundle has no flags.

`ready/settings/` is not a primitive scope. It holds git-tracked workspace and
app settings that help tools manage the Ready tree, such as product-agent
context preferences, view settings, or local app defaults that should travel
with the repo. Settings files should use explicit settings schemas, avoid raw
secrets, and not duplicate product truth that belongs in primitives,
governance, artifacts, or flags. Runtime artifacts, build outputs, uploaded
product data, generated product manifests, runtime cache, user-private
preferences, and provider secrets do not belong in `ready/settings/`.

## Record kinds and primitive classes

Ready records use `kind` to identify the object family:

- `primitive`: durable bundle input required to build, prove, operate, or
  govern the intended product bundle.
- `flag`: temporary bundle-work record for unresolved Ready-bundle decisions,
  proposed changes, blockers, discrepancies, readiness gates, and proof gaps.
- `governance`: bundle management record for authority, process, policy, or
  agent behavior.
- `setting`: git-tracked workspace or app configuration used by Ready tools.
- `manifest`: structural bundle metadata.

Primitive records use `class` for their product family:

- `premise`: why the product or feature should exist.
- `intent`: what the product promises to do; user-facing views may call an
  intent an implementation bundle.
- `standard`: how the product must be built, measured, judged, or maintained.
- `service`: external or separately maintained things the product depends on to
  build, prove, run, or monitor.
- `artifact`: durable supporting content such as source documents, proof
  corpora, snippets, designs, and manifests.

`type` is the narrower subtype under a class. Keep primitive classes stable and
use subtypes from `ready/standard/vocabulary.yaml` to make the discovery process
intuitive. Constraints are standards: use `class: standard` records for
constraints, rules, quality bars, compliance needs, measurement rules, and
non-negotiable behavior.

Flags use X-family ids but are not primitives. Governance, settings, and
manifest records also have their own kinds. Product compilers should use
explicit `kind`, `class`, `type`, schema, and path metadata rather than assuming
a directory name gives every record the same authority.

## Governance and artifacts

Project-specific governance documents live under `ready/governance/`.
Reusable Ready resources, task-skill bundles, and templates live in the Ready
standard bundle. Together they constrain how product truth is created,
changed, approved, and handed to agents.

Artifacts that support product truth usually live under `ready/product/artifacts/`.
Flags may link to those artifacts or keep their own small workflow artifacts
under `ready/flags/` when the artifact only supports the temporary flag.
Artifact descriptors are primitives with `class: artifact`. They support
product logic and proof with samples, snippets, design assets, manifests,
screenshots, and handoff notes. Attached payload files can use their native
formats when they are clearly referenced by a descriptor. Each descriptor has
one purpose; files with different purposes need different descriptors.

Ready-carried artifact payloads are source material for implementation and
proof. If a payload is meant to persist in the implemented product, the
orchestrator must copy, transform, or compile it into product-owned code,
assets, storage, or build inputs. The product copy is the runtime/build artifact;
the Ready descriptor remains the source-of-intent and traceability record.

## Generated views

Compiled docs, resolved coding handoffs, designer briefs, PM summaries,
customer-facing explanations, and app views are projections of the product tree.
They can be cached or committed only when marked generated and traceable to
source records.

A human-authored document may propose a product change, but the accepted change
must be applied back to the product tree.

Apps may compile a resolved target-state specification from product primitives,
standards, services, artifacts, and any relevant flags. Human views may
highlight differences from code, evidence, or prior commits, but coding
handoffs should provide the resolved intended state plus proof gates, not just
stored differences.

## Non-goals

The product tree is not a backend replacement forever. It is the first durable
source of truth. A backend can later index it, render it, coordinate live team
state, and host agents without changing the core product model.

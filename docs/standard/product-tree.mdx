# Product Tree Standard

A product tree is a committed filetree that stores product truth in typed
records. Primitive source records use `.ready.yml`: plain YAML with a distinct
extension so importers know exactly how to parse it. The tree should behave like
a healthy codebase: fragmented enough to review, explicit enough to validate,
and structured enough to compile into different views.

A Ready tree on a branch represents the intended product state for that branch.
It should say what should exist, not how to change the current implementation.
Git history records how product truth evolved; Ready primitives describe the
currently intended state.

## Source of truth

Each primitive owns its durable truth in one `.ready.yml` file:

- The YAML document is the machine contract.
- Apps and docs compile reader-friendly views from the YAML.
- Stable ids are the durable identity.
- Paths and titles are refactorable.
- Generated views derive from the tree.

Markdown is still useful for navigation, governance, and generated views, but it
is not the primitive source format. Top-level index files are allowed for
navigation, but they must not duplicate record truth.

Hypothetical ideas and concepts should usually be modeled as alternate Ready
branches or worktrees rather than speculative fragments inside the canonical
branch. A decision or flag can point to that branch when the team needs to choose
or coordinate, but the idea's product shape should remain a coherent intended
state tree.

## Tree directories

The root `ready/manifest.yaml` owns the product, flags, governance, settings,
and standard package locations. The standard does not define additional
product-state overlays. A Ready tree version should state one coherent intended product; Git
commits, branches, and worktrees carry history, alternatives, and hypothetical
concepts.

Use `product_directory`, `flags_directory`, `governance_directory`, and
`settings_directory` in the root manifest:

```yaml
product_directory: ready/product
flags_directory: ready/flags
governance_directory: ready/governance
settings_directory: ready/settings
```

`ready/product/` is the canonical source of intended product truth for the
current tree version. It holds premises, intents, standards, services, product
artifacts, proof corpora, and other records that define what should exist.

`ready/flags/` is a peer workflow directory. It may hold Decision flags,
blockers, discrepancy notes, version-review notes, and claim-coordination
records. Flags can point at product truth, but they do not own the product
truth and they do not replace intent Completion Proof.

`ready/settings/` is not a primitive scope. It holds git-tracked workspace and
app settings that help tools manage the Ready tree, such as product-agent
context preferences, view settings, or local app defaults that should travel
with the repo. Settings files should use explicit settings schemas, avoid raw
secrets, and not duplicate product truth that belongs in primitives,
governance, artifacts, or flags. Runtime cache, user-private preferences, and
provider secrets do not belong in `ready/settings/`.

## Primitive classes

A Ready primitive is a typed, addressable, reusable unit of product knowledge,
decision state, workflow state, evidence, or governance context. "Primitive"
does not mean every record has the same compiler authority. Tools should read a
record's class, type, path, and role before deciding whether it is product truth,
workflow state, evidence, generated context, or governance.

Core classes:

- `product_logic`: source product truth. Common types are `premise`, `intent`,
  `standard`, and `service`.
- `decision_workflow`: Decision flags, blockers, drift, proof gaps, and other
  records that direct attention or readiness.
- `evidence_artifact`: resources, proof corpora, snippets, designs, manifests,
  screenshots, and handoff notes. The primitive is the artifact descriptor; a
  bulky, binary, private, or generated payload file is an attachment or safe ref,
  not the semantic product reference.
- `governance`: project authority, process, agent guidelines, skills, and
  templates that constrain how the tree may be changed.

The first product-logic primitive types are:

- `premise`: why the product or feature should exist.
- `intent`: what the product promises to do.
- `standard`: how the product must be built, measured, judged, or maintained.
- `service`: what the product depends on to build, prove, run, or monitor.

Keep these four primitive types stable and use subtypes from
`ready/standard/vocabulary.yaml` to make the discovery process intuitive.
Constraints are standards: use `standard` records for constraints, rules,
quality bars, compliance needs, measurement rules, and non-negotiable behavior.

Decision flags and flags are decision/workflow primitives and use X-family ids.
Artifact descriptors are evidence/artifact primitives and use A-family ids.
Governance docs are governance primitives. Product compilers should use
explicit class and role metadata rather than assuming a directory name gives
every primitive the same authority.

## Governance and artifacts

Project-specific governance documents live under `ready/governance/`.
Reusable Ready resources, task-skill packages, and templates live in the Ready
standard package. Together they constrain how product truth is created,
changed, approved, and handed to agents.

Artifacts that support product truth usually live under `ready/product/artifacts/`.
Flags may link to those artifacts or keep their own small workflow artifacts
under `ready/flags/` when the artifact only supports the temporary flag.
Artifact descriptors are evidence/artifact primitives that support product
logic and proof with samples, snippets, design assets, manifests, screenshots,
and handoff notes. Attached payload files can use their native formats when they
are clearly referenced by a descriptor. Each descriptor has one purpose; files
with different purposes need different descriptors.

## Generated views

Compiled docs, resolved coding handoffs, designer briefs, PM summaries,
customer-facing explanations, and app views are projections of the product tree.
They can be cached or committed only when marked generated and traceable to
source records.

A human-authored document may propose a product change, but the accepted change
must be applied back to the product tree.

Apps may compile a resolved target-state specification from product primitives,
standards, services, artifacts, and any relevant flags. Human views may
highlight differences from code, evidence, or prior commits, but coding
handoffs should provide the resolved intended state plus proof gates, not just
stored differences.

## Non-goals

The product tree is not a backend replacement forever. It is the first durable
source of truth. A backend can later index it, render it, coordinate live team
state, and host agents without changing the core product model.

# `.ready` Bundle

A `.ready` bundle is the committed source bundle an orchestrator reads to
build, prove, operate, or govern the intended product. A bundle should be
complete enough that a coding orchestrator can rebuild the product cleanly from
the `.ready` tree, without depending on unstated tickets, chat memory, or
current code structure.

Git branches and worktrees carry alternate bundle versions. A branch should
state one coherent intended product state.

## Bundle Contents

A Ready bundle includes:

- product primitives under `ready/product/`;
- artifact primitives and payload refs needed to understand, build, prove, or
  review the product;
- project governance records under `ready/governance/`;
- tracked settings under `ready/settings/` when tools need them;
- manifests that describe bundle structure;
- the vendored Ready Standard bundle under `ready/standard/`;
- flags under `ready/flags/` only while unresolved bundle decisions, proposed
  changes, blockers, discrepancies, readiness gates, or proof gaps keep the
  bundle incomplete, disputed, divergent, or not safely claimable.

Governance travels with the bundle so humans and agents share the same
authority, access, and process versions. It constrains who may change the Ready
files and how bundle work is approved; it is not product runtime
implementation truth.

Flags are not bundle truth. They are temporary bundle-work records that
explain why the bundle is not yet complete, why a version diverges, or what
potential bundle change is being discussed. A completed bundle has no flags.

## Completeness

A complete Ready bundle has:

- one coherent intended product state;
- no flags;
- no status-bearing records in `draft`;
- no status-bearing records in `blocked`;
- ready intents, also called implementation bundles in user-facing views, with
  concrete Completion Proof;
- standards, services, artifacts, proof corpora, design artifacts,
  service access, setup references, and verification commands represented;
- sensitive, private, bulky, or external materials represented by safe refs
  owned by artifact or service records;
- bundle manifests and governance records sufficient for an orchestrator to
  know what to read and what authority rules apply.

A bundle can be useful before it is complete. Incomplete bundles can support
discovery, review, planning, simulation, partial implementation, and proof work,
but a complete product should not be expected from an incomplete bundle.

## Relationship To The Product

The `.ready` tree is the product's source-of-intent bundle, not the product's
runtime artifact store. The tree may exist before any implementation exists. A
fully implemented product must also be able to build and run without reading the
`.ready` tree at runtime.

The `.ready` tree may carry source artifacts that are intended to become
persistent product artifacts, such as seed assets, templates, snippets, design
files, proof fixtures, manifests, or example payloads. During implementation, an
orchestrator copies, transforms, or compiles those Ready-carried artifacts into
the product repository or product-owned storage. The implemented product owns
those runtime/build copies after the build. It must not depend on
`ready/product/`, `ready/flags/`, or `ready/settings/` as a runtime asset path.

Use Ready artifact descriptors to explain the artifact's provenance, authority,
intended use, proof role, and expected product destination. Do not use
`ready/settings/` to store runtime assets, generated runtime files, uploaded
product data, product-owned manifests, or other implementation artifacts.
Settings only configure tools that manage the Ready tree.

## Orchestrator Rules

Before implementation, an orchestrator should inspect bundle completeness.

- If any status-bearing record is `draft`, warn that the build is from an
  incomplete bundle and should not be expected to produce a complete product.
- If any status-bearing record is `blocked`, do not
  implement unless the user or governing process gives explicit override
  instructions.
- If flags exist, treat the bundle as incomplete or intentionally divergent.
  Flags can hold proposed bundle-change discussion, version-divergence logic,
  blockers, decisions, discrepancies, readiness gates, or proof gaps, not edit
  instructions.
- If an override is granted, preserve the override, the unresolved gaps, and the
  limits of the resulting build evidence.
- Build from the resolved intended product state, standards, services,
  artifacts, proof, governance, and applicable flag closure criteria. Do not
  build only from diffs or deltas.
- When Ready-carried artifacts are needed by the implemented product, copy,
  transform, or compile them into product-owned paths or storage and preserve
  traceability back to the Ready artifact descriptor. Do not leave runtime
  product behavior dependent on reading the Ready tree.

## Status Semantics

Use simple status values consistently:

- `ready`: resolved for its record kind. Product primitives compile as current
  intended truth. Flags do not use `ready` status.
- `draft`: incomplete or not yet approved for its record kind; compile only
  with warning.
- `blocked`: cannot be built, proven, operated, or claimed without resolving
  the blocker or receiving explicit override instructions.

Use these statuses consistently wherever a Ready source record carries
top-level `status`.
